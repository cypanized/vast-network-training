How do multiple subnets work without VLAN tags? (Lab setup)
In this lab environment, VIPs from different subnets (172.200.x/16, 10.10.x/16, 192.168.x/24) are all assigned as secondary IP addresses on the same physical interface without VLAN tagging. This is a simplified configuration for testing purposes.

1. Multiple IPs on one interface: Linux allows adding many IP addresses from different subnets to a single interface (ip addr add). Each VIP is simply another address on enp65s0f0 or enp65s0f1.
2. L3 routing, not L2 segmentation: In the lab, the upstream switch routes all VIP traffic on the same untagged (native VLAN) port — routing packets based on the destination IP, not a VLAN tag.
3. Shared broadcast domain: All subnets share the same Layer 2 broadcast domain on the physical link. ARP requests for any VIP are answered by the interface regardless of subnet.
4. Upstream switch: SVI / IRB interfaces: The upstream leaf or aggregation switch must have an SVI (Switch Virtual Interface, Cisco/Arista) or IRB (Integrated Routing and Bridging, Juniper/NVIDIA) for each client subnet. These are Layer 3 gateway interfaces that bridge between the VLAN carrying the VAST traffic and the IP routing table. For example, a switch would have interface Vlan100 with IP 172.200.0.1/16 and interface Vlan200 with IP 10.10.0.1/16 — each acting as the default gateway for that subnet. The SVI/IRB performs proxy-ARP or routes traffic destined for VAST VIPs down to the leaf ports. In an EVPN-VXLAN fabric, the IRB can be a distributed anycast gateway on every leaf, so client traffic is routed at the nearest switch rather than hair-pinning through a central router.

In production / customer environments: Customers typically use VLANs to segregate client subnets — each subnet gets its own VLAN ID and the VAST front-end ports are configured as trunk ports carrying tagged traffic. This provides proper Layer 2 isolation between tenants or workloads, better security, and aligns with enterprise network standards. The choice depends on the customer's network design, security requirements, and whether multi-tenancy isolation is needed.

How do multiple subnets work without VLAN tags? (Lab setup)
In this lab environment, VIPs from different subnets (172.200.x/16, 10.10.x/16, 192.168.x/24) are all assigned as secondary IP addresses on the same physical interface without VLAN tagging. This is a simplified configuration for testing purposes.

1. Multiple IPs on one interface: Linux allows adding many IP addresses from different subnets to a single interface (ip addr add). Each VIP is simply another address on enp65s0f0 or enp65s0f1.
2. L3 routing, not L2 segmentation: In the lab, the upstream switch routes all VIP traffic on the same untagged (native VLAN) port — routing packets based on the destination IP, not a VLAN tag.
3. Shared broadcast domain: All subnets share the same Layer 2 broadcast domain on the physical link. ARP requests for any VIP are answered by the interface regardless of subnet.
4. Upstream switch: SVI / IRB interfaces: The upstream leaf or aggregation switch must have an SVI (Switch Virtual Interface, Cisco/Arista) or IRB (Integrated Routing and Bridging, Juniper/NVIDIA) for each client subnet. These are Layer 3 gateway interfaces that bridge between the VLAN carrying the VAST traffic and the IP routing table. For example, a switch would have interface Vlan100 with IP 172.200.0.1/16 and interface Vlan200 with IP 10.10.0.1/16 — each acting as the default gateway for that subnet. The SVI/IRB performs proxy-ARP or routes traffic destined for VAST VIPs down to the leaf ports. In an EVPN-VXLAN fabric, the IRB can be a distributed anycast gateway on every leaf, so client traffic is routed at the nearest switch rather than hair-pinning through a central router.

In production / customer environments: Customers typically use VLANs to segregate client subnets — each subnet gets its own VLAN ID and the VAST front-end ports are configured as trunk ports carrying tagged traffic. This provides proper Layer 2 isolation between tenants or workloads, better security, and aligns with enterprise network standards. The choice depends on the customer's network design, security requirements, and whether multi-tenancy isolation is needed.

For example, a switch would have interface Vlan100 with IP 172.200.0.1/16 and interface Vlan200 with IP 10.10.0.1/16 — each acting as the default gateway for that subnet. The SVI/IRB performs proxy-ARP or routes traffic destined for VAST VIPs down to the leaf ports. In an EVPN-VXLAN fabric, the IRB can be a distributed anycast gateway on every leaf, so client traffic is routed at the nearest switch rather than hair-pinning through a central router.

In production / customer environments: Customers typically use VLANs to segregate client subnets — each subnet gets its own VLAN ID and the VAST front-end ports are configured as trunk ports carrying tagged traffic. This provides proper Layer 2 isolation between tenants or workloads, better security, and aligns with enterprise network standards. The choice depends on the customer's network design, security requirements, and whether multi-tenancy isolation is needed.

B2B IPMI

Back-to-back IPMI daisy-chains each server's BMC port to the next server's GbE1, forming a loop for out-of-band management without a dedicated switch. Also covers management switch and SSH access topologies.

Note: This diagram shows the B2B loop when management is outband — IPMI uses the dedicated BMC port daisy-chained via GbE1.

How B2B IPMI Works

In a back-to-back (B2B) IPMI configuration, each server's BMC port is directly cabled to the GbE1 port of the next server in the rack. The last server loops back to the first, forming a daisy-chain ring.

Advantages

  • No switch required — zero cost
  • Simple cabling between adjacent servers
  • Self-contained within the rack
  • Works with any number of servers

Considerations

  • Must SSH hop through servers to reach BMC
  • If a server in the chain is down, chain breaks
  • Requires proper IP planning (same subnet)

IPMI via Management Switch

Each server's shared GbE1 port (carrying both OS IP and IPMI IP) connects to a simple unmanaged switch. An admin laptop also connects to the same switch, giving direct access to all BMCs and OS interfaces without SSH hopping.

Advantages

  • Direct access to every BMC — no SSH hops
  • No dependency on server OS being up
  • Simple unmanaged switch is sufficient
  • Easy to scale — just add ports

Considerations

  • Requires an additional switch (small cost)
  • Extra cabling to the switch
  • Switch is a single point of failure for mgmt
  • Shared GbE port carries OS + IPMI traffic

Accessing IPMI via SSH

In a B2B configuration, to reach a BMC you SSH into the server whose GbE1 is connected to that BMC, then use ipmitool locally. The admin laptop connects to the high-speed management network and hops through to reach the IPMI chain.
